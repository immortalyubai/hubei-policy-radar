# 架构与取舍

## 为什么不直接把 `policy_crawling` 当成最终主体

`policy_crawling` 适合作为政府网站适配器参考：已有 FastAPI、Vue、MySQL、定时任务、附件处理和若干省级页面规则，能加速第一个官网抓取器。但它的核心记录以单一网页为中心，无法自然表达“同一事项由公众号先发、官网后发”的多来源证据关系，且现有去重、测试、安全默认值和湖北覆盖都不足以直接作为产品底座。

本项目采用组合方案：

- 保留其“按来源写适配器”的思路，只迁移必要的页面规则。
- 重新建设统一数据核心和网站。
- 使用 `items` 表示归并后的事项，`item_sources` 表示公众号、官网等证据。
- 官网证据到达后，将 `pending_official` 升级为 `official_verified`，而不是新增重复政策。

## 分层

1. **来源层**：政府列表、公众号单篇、未来的 RSS 与人工导入。
2. **标准化层**：标题、地区、发布单位、发布日期、截止日期、适用对象、收益、主题。
3. **筛选层**：关键词、事项类型、时效、资金信号、重点产业与来源优先级。
4. **证据层**：严格键去重、多来源归并、公众号待核验、冲突保留。
5. **展示层**：政策流、详情、来源状态和 JSON API。

## 自动合并边界

按顺序匹配：

1. 同来源外部 ID 或相同规范 URL。
2. 同地区、相同文号。
3. 地区 + 规范标题 + 发布单位 + 发布日期的严格键。
4. 标题近似只生成候选，不能自动合并。

公众号和官网的截止日期、适用对象或正文关键条件冲突时，后续应标记 `conflict`，不得静默覆盖。

## 调度

- 每两小时：重点来源第一页，低并发扫描。
- 每天夜间：前三页补扫，发现连续已存在记录后停止翻页。
- 详情页只抓新条目；429/5xx 有限重试，403/412 记录来源异常后退出。
- 湖北省政府主站如遇 JavaScript 挑战，应由独立采集器有限回退到真实浏览器，不在网站 Worker 内运行浏览器。
